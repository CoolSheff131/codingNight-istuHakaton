## Преамбула

В университете муми-долла учатся три группы: Муми-тролли, Снорки и Мумрики. Учатся себе да учатся, занятия проводят преподаватели, сами занятия проводятся на пяти разных локациях. Иногда занятия переносятся, иногда проходят мероприятия, по понедельникам все дружно ходят на проектную деятельность.
И вот как-то им все не хватает возможности в удобном формате посмотреть свое расписание.
Сегодня мы предлагаем вам разработать собственное решение для просмотра расписания и облегчить жизнь зверьками муми-долла

## Требования к приложению

Минимальное работающее решение должно давать возможность:
просматривать расписание в понедельном формате
просматривать расписание для конкретной группы
просматривать расписание для конкретного преподавателя
просматривать расписание для конкретной аудитории
давать возможность в удобной форме переключаться между этими расписаниями
выводить проекты со всеми его участника во всех расписаниях групп/аудиторий/преподавателей принимающих в нем участие
выводить мероприятие со всеми его участника во всех расписаниях групп/аудиторий/преподавателей принимающих в нем участие
если пара перенесена на другую пару либо на другой день, это надо как-то показать

## Требования к архитектуре и технологиям

Вы можете использовать любые технологии которые вам удобны.

Строгих требований к архитектуре нет. В идеале выбрать одну из следующих моделей

Чистое бэкэнд приложение, которое рендерит шаблоны страниц по данным из БД
Бэкэнд, который предоставляет API для выдачи данных из БД + фронтенд в виде SPA
Фронтенд в виде SPA без бэкенда
В первую очередь интересны именно UI/UX решения

Любые новые идеи приветствуются. Если вам кажется что в наших данных чего-то не хватает, смело добавляйте. Самое важное чтобы костяк расписания не менялся. И все имеющиеся данные были видны в итоговом формате.

## Правила ведения проекта

Команды могут приступать к работе непосредственно после получения задания.
Работа команд над проектом осуществляется в любом удобном для участников месте и любое время.
Работа над проектами ведется до 26 мая 2022 года включительно, в конце каждого дня - до 23:59 - необходимо фиксировать все результаты работы в свой удаленный репозиторий.

## Данные расписания скачать (http://hackaton2022.ru/schedule.json)

Формат данных:
{
"id": 4376056, // id занятия
"type": 3, // тип, 1 -- лекция, 2 -- практика, 3 -- лаба
"day": 6, // день, от 1 до 7, 1 -- понедельник, 2 -- вторник и т.д.
"pair": 6, // пара, от 1 до 8
"week_begining": "2022-04-11", // дата начала недели
"groups": ["Муми-тролли"], // группы, присутствующие на занятии
"subgroup_number": 1, // номер подгруппы, если занятие имеет тип 3, то в groups будет ровно одна группа,
// а 1 означает, что на занятие придет только первая половина группы
// 2 означает, что на занятие придет только вторая половина группы
"discipline": "Собирание оригами", // название дисциплины
"teachers": ["Шуссель"], // преподаватели, которые ведут занятия
"auditories": ["Обсерватория"] // аудитории, задействованные для занятия
}

## Данные мероприятий, переносов и проектов скачать (http://hackaton2022.ru/queries.json)

Формат данных:
{
"id": 25585, // id события
"dt": "2022-04-16", // дата проведения
"pair": 8, // номер пары, от 1 до 8
"description": "Наблюдение за кометой", // название
"groups": ["Снорки", "Муми-тролли", "Мумрики"], // список участвующих групп
"teachers": ["Муми-папа"], // список участвующих преподавателей
"auditories": ["Обсерватория"], // список задействованных аудиторий
"type": 0, // тип, 0 -- мероприятие, 1 -- проект, 3 -- перенос пары, 4 -- перенесенная пара
"related_queries": [], // если тип 3 или 4, тут будет ссылка на связанные с переносом id событий
"affected_schedule_id": null // id занятия, которое перенеслось
}

## Регламент презентации проектов

В заключительный день (27 мая 2022 г) проводится жеребьевка для определения порядка представления результатов работы команд.
Каждой команде предоставляется 10 минут, в течение которых они могут в любой форме презентовать полученные результаты (например, в виде мультимедиа-презентации и/или непосредственной демонстрации).
После представления доклада члены команды отвечают на вопросы жюри хакатона по своему проекту.
Каждый член жюри оценивает результаты по десятибалльной шкале, и суммарная оценка фигурирует в публикуемых итогах хакатона.
Выделяется три призовых места, решение о их присуждении выносится жюри.
Жюри оценивают представленные проекты

## Критерии оценки жюри

Работа команды – аккуратность коммитов, разделение обязанностей;
Защита и представление проекта (лаконичность изложения, оформления и ясность представления);
Техническая оценка реализации: (обоснованность выбора технологий, грамотность их применения, проработанность архитектурного решения);
UX-решения: функциональность, удобство в использовании;
Степень проработанности всего проекта: завершенность, глубина проработки, охват реализованного решения тестами;
Ответы на вопросы членов жюри.
